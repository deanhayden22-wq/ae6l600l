;   Max raw value = 0x2B (43) -> minimum AFR = 11.00
;
; Failsafe Map (@ 0xD05C4):
;   Max raw value = 0x46 (70) -> minimum AFR = 9.50
;   At 2000 RPM / 2.60 g/rev: raw = 0x3C (60)
;     AFR = 14.7 / (1 + 60 * 0.0078125) = 14.7 / 1.46875 = 10.01 ~ 10.02
;   This is the root cause of observed "10.02 AFR Final Fueling Base".

AFC / CLOSED-LOOP FUEL FEEDBACK SYSTEM — COMPLETE TRACE
================================================================================
ROM: AE5L600L (2013 USDM Impreza WRX MT)  |  MCU: SH7058 SH-2A BE
Analysis: Ghidra 12.0.2 + binary cross-reference tracing
================================================================================

EXECUTIVE SUMMARY
=================
There is NO single table or function called "AFC" in this ECU. The Subaru
closed-loop fuel system is decomposed into 5 subsystems:

  1. CL/OL State Machine    — Decides when to run closed-loop vs open-loop
  2. Front O2 Sensor Input   — Wideband lambda sensor reading → AFR
  3. Short-Term Correction    — Real-time AFC adjustment (not yet labeled in Ghidra)
  4. A/F Learning (LTFT)      — Long-term fuel trim stored in 4 airflow ranges
  5. AF 3 Correction          — Rear O2 trim (DISABLED in this ROM — limits = 0%)

The code lives in ROM 0x33000-0x37000 and is dispatched from two secondary
task tables (NOT the 59-entry periodic scheduler used by knock/FLKC).


================================================================================
SECTION 1: DISPATCH — HOW THE FUELING CODE GETS CALLED
================================================================================

The fueling functions are invoked from two dispatch tables in the scheduler
region, separate from the 59-task periodic table @ 0x4AD40:

  DISPATCH TABLE A @ 0x480B8 (8 fueling entries):
  ─────────────────────────────────────────────────
  0x480B8: -> 0x33278   Main fueling pre-calc
  0x480CC: -> 0x34488   A/F Learning sub-dispatcher
  0x480D0: -> 0x357D0   CL/OL transition sub B
  0x480D4: -> 0x34EC8   A/F Learning airflow range processor
  0x480D8: -> 0x36008   CL/OL delay manager A
  0x480DC: -> 0x36A98   CL/OL hysteresis handler
  0x480E0: -> 0x36BF4   CL/OL delay manager B
  0x480E4: -> 0x36E60   Post-transition handler

  DISPATCH TABLE B @ 0x4A0B8 (6 fueling entries):
  ─────────────────────────────────────────────────
  0x4A0B8: -> 0x332A2   Main fueling entry (NON-RETURNING)
  0x4A0E0: -> 0x36C3C   OL/CL state cleanup
  0x4A0E8: -> 0x3452A   A/F Learning core (refs CC064)
  0x4A0EC: -> 0x34EF4   A/F Learning airflow update
  0x4A0F0: -> 0x3580C   CL/OL transition core
  0x4A0F4: -> 0x3605E   OL fuel map selector (refs CC16C)

  The main fueling entry at 0x332A2 is flagged as "non-returning" by Ghidra,
  suggesting it's a persistent loop or state machine rather than a one-shot task.


================================================================================
SECTION 2: CL/OL STATE MACHINE — WHEN DOES AFC RUN?
================================================================================

AFC (closed-loop correction) only runs when the ECU is in Closed Loop (CL) mode.
The CL/OL transition is a multi-condition state machine controlled by ~15
calibration tables. The transition logic occupies ROM 0x35100-0x36C00.

─── CL → OL Transition (AFC gets disabled) ───────────────────────────────

The ECU transitions from CL to OL when ANY of these conditions clears the
CL delay counter to zero:

  CONDITION                          TABLE ADDRESS    VALUE IN THIS ROM
  ───────────────────────────────────────────────────────────────────────
  RPM >= Max (Per Gear)              0xCC180          3200-3700 RPM (gear-dep.)
  RPM >= Max (Neutral)               0xCC1A8          6000-6100 RPM
  Throttle >= Max                    0xCC1D8          37.9 / 90.0 deg
  Vehicle Speed >= Max               0xCC1F4          141.7 / 144.9 km/h
  Engine Load >= Max                 0xCC204          0.95 / 1.10 g/rev
  ECT < Minimum                      0xCC17C          -12.0 °F

  CRITICAL: CL to OL Delay @ 0xCBC62 = 0.0
  ──────────────────────────────────────────
  The base CL→OL delay counter is SET TO ZERO in this ROM.
  This means there is NO delay — the transition to OL is IMMEDIATE when
  any throttle or BPW threshold is exceeded. The SI-DRIVE Intelligent
  delay (0xCBC5C) is also 0.0.

  When the delay IS non-zero, the transition uses:
    - Throttle threshold:  CL→OL Transition with Delay (Throttle) @ 0xCCD78
    - BPW threshold:       CL→OL Transition with Delay (BPW) @ 0xCE5F8
    - Counter increment:   CL→OL Counter Step Value (MAF) @ 0xCE640

  The counter increments each cycle that throttle OR BPW exceeds the threshold.
  When counter >= delay value, transition occurs.

─── OL → CL Transition (AFC gets re-enabled) ─────────────────────────────

  The ECU returns to CL when:
    Throttle <= (CL→OL Throttle Threshold - Hysteresis)
    AND
    BPW <= (CL→OL BPW Threshold - BPW Hysteresis)

  Hysteresis values:
    Throttle hysteresis:  8.4 deg  (0xCC178)
    BPW hysteresis:       756.0    (0xCC174)

  NOTE: The large BPW hysteresis (756) means the ECU requires a significant
  drop in fuel demand before returning to CL mode.

─── Code Structure ────────────────────────────────────────────────────────

  Literal pool clusters in the CL/OL transition code:

  0x35120-0x351D8 (46 entries) — Initial CL/OL check
    Reads: RPM (FFFF6350), MAF (FFFF6624), ECT (FFFF63CC),
           Gear (FFFF5E94), Vehicle Speed
    Refs:  CC0AC-CC0D0 calibration, multiple table processor calls

  0x36278-0x362F8 (31 entries) — OL fuel map switch (IAM check)
    Key ref: 0xCC16C (IAM threshold = 0.5)
    Reads:  IAM value (FFFF3234), RPM, MAF
    Calls:  0x22F92 (utility function)
    Logic:  If IAM < 0.5 → switch to failsafe fuel maps

  0x36534-0x365D8 (41 entries) — Per-gear RPM limit check
    Refs:  CC180-CC1D4 (all 10 per-gear RPM thresholds)
    Reads: Current gear (FFFF5E94/5E92/5E93/5DB5)
    Logic: If RPM >= threshold for current gear → clear CL delay → OL

  0x367B8-0x36824 (27 entries) — Multi-condition CL delay clear
    Refs:  CC1A8 (Neutral RPM), CC1D8 (Throttle max), CC1F4 (Speed max),
           CC204 (Load max), CC17C (ECT min), CBC5A (Load counter thresh),
           CBC5C/CBC62 (CL/OL delay values), CBC60/CBC64
    Logic: Checks each condition sequentially, clears delay if exceeded

  0x36A0C-0x36A98 (33 entries) — Hysteresis and OL entry confirmation
    Refs:  CC178 (Throttle hysteresis), CC174 (BPW hysteresis)
    Reads: FFFF798C, FFFF79F8 (current throttle/BPW state)
    Logic: Applies hysteresis to prevent CL/OL oscillation


================================================================================
SECTION 3: FRONT O2 SENSOR — THE FEEDBACK INPUT
================================================================================

The front O2 sensor is a WIDEBAND (air/fuel ratio) sensor, not a narrowband
switching sensor. It reads pump current in milliamps and converts to AFR.

─── Sensor Scaling Table @ 0xD8DA8 (Y-axis @ 0xD8D74) ───────────────────

  Sensor Current (mA)   →   Lambda    →    AFR
  ────────────────────────────────────────────────
  -1.300 mA             →   0.7586    →   11.15
  -0.870 mA             →   0.8276    →   12.17
  -0.470 mA             →   0.8966    →   13.18
  -0.140 mA             →   0.9655    →   14.19
   0.000 mA             →   1.0000    →   14.70  ← stoichiometric
   0.100 mA             →   1.0345    →   15.21
   0.240 mA             →   1.1034    →   16.22
   0.390 mA             →   1.1724    →   17.23
   0.530 mA             →   1.2414    →   18.25
   0.640 mA             →   1.3103    →   19.26
   0.740 mA             →   1.3793    →   20.28

  Rich limit:   AFR 11.02 (lambda 0.750) @ 0x21CAC
  Lean limit:   AFR 29.40 (lambda 2.000) — hardcoded in compensation formula

  Atmospheric compensation @ 0xC3708:
    Formula: ((Front O2 AFR - 14.7) × Compensation) + 14.7
    The compensation value adjusts for altitude/barometric pressure.

  Front AF Sensor Smoothing (Alpha category):
    A 4×4 table indexed by Smoothed MAF × Engine Load.
    WARNING: "DO NOT SET TO 100% or higher!"
    Lower values = less smoothing = faster response.
    This smoothing factor directly affects how quickly AFC can respond.

─── Sensor Processing Path (Code Functions) ───────────────────────────────

  0x004A2C:  frontO2_scaling_init
             Initializes/reads front O2 sensor scaling via descriptor @ 0xAF45C.

  0x058902:  frontO2_scaling_lookup
             Performs the 2D interpolation through the scaling table.
             Accesses descriptor @ 0xAF468 (Y-axis at D8D74, data at D8DA8).

  0x021A40:  frontO2_process (MAIN O2 PROCESSING FUNCTION)
             - Loads rich limit (lambda 0.750) from inline literal pool at 0x21CAC
               via mova @(0x230,PC),R0 + fmov.s @R0,FR12
             - Accesses atmospheric compensation descriptor @ 0xAAE8C
             - Applies formula: ((AFR - 14.7) × compensation) + 14.7
             - Clamps result to rich limit

  0x01FE54:  frontO2_comp_atm
             Atmospheric pressure compensation sub-function.
             Accesses descriptor @ 0xAAE78.

  Processing pipeline:
  1. ADC reads wideband sensor pump current → raw mA value
  2. frontO2_scaling_init (0x4A2C) → initializes scaling
  3. frontO2_scaling_lookup (0x58902) → 2D interpolation → lambda value
  4. frontO2_process (0x21A40) → rich limit clamp + atm compensation
  5. Smoothing filter applied (Alpha table)
  6. Result stored in RAM for AFC feedback loop


================================================================================
SECTION 4: A/F LEARNING — LONG-TERM FUEL TRIM (LTFT)
================================================================================

The A/F Learning system stores long-term corrections in 4 airflow ranges.
Code is at ROM 0x34490-0x34EC8, dispatched from 0x4A0E8 (entry: 0x3452A).

─── Calibration Data ─────────────────────────────────────────────────────

  A/F Learning #1 Limits @ 0xCC064:
    Minimum: -0.250 (= -25%)
    Maximum: +0.250 (= +25%)

  A/F Learning #1 Airflow Ranges @ 0xCC074:
    Range A:  6.0 - 23.0 g/s    (idle / light cruise)
    Range B: 40.0 - 80.0 g/s    (moderate load)
    Range C:  0.95 - 1.05 g/s   (deceleration / very low flow)
    Range D: 35.0 -  0.0 g/s    (DISABLED — max < min)

  A/F #1 Learning Max Threshold (Alpha): Controls the maximum MAF where
    learning values are applied. Setting below D range prevents fuel
    learnings from carrying into open loop fueling.

  A/F #1 Learning Min Threshold (Alpha): Controls minimum MAF for learning.

─── Code Analysis ─────────────────────────────────────────────────────────

  Entry point: 0x3452A (called from dispatch table B @ 0x4A0E8)

  Function structure (traced from disassembly):

  0x3452A: Main A/F Learning entry
    Saves r9, r13, r14, PR
    Calls:
      0x34D52  — Subroutine (purpose TBD, appears to be precondition check)
      0x345A4  — CL Active Check (see below)
      0x34778  — Subroutine (learning range selection?)
      0x34884  — Subroutine (learning value update?)
      0x34B1C  — Subroutine (learning value storage?)
      0x34CC8  — Post-processing
      0x34C18  — Cleanup
    Reads RAM: FFFF7D68 (float), FFFF78A4 (output area)
    Writes: FFFF78A0 region (A/F learning output data structure)

  0x345A4: CL Active Check function
    This function checks 8 conditions to determine if CL is active.
    Returns 1 (CL active, learning OK) or 0 (not active, skip learning).

    Conditions checked (all must pass for CL active):
      1. RAM[FFFF8F24] — Global enable flag
      2. Comparison vs calibration @ 0xCC020 (= 70.0) — likely MAF threshold
         FFFF78A0[-20] (float) must be <= 70.0 g/s
      3. RAM[FFFF73A4] — via BE608 table processor check
      4. RAM[FFFF7354] — via BE608 table processor check
      5. RAM[FFFF7374] — via BE608 table processor check
      6. RAM[FFFF7A14] — via BE608 table processor check
      7. RAM[FFFF7A20] — via BE608 table processor check
      8. RAM[FFFF7D18] — Scheduler status (also used by FLKC)
      9. r14 (FFFF8F24 flag again)
     10. RAM[FFFF7BE2] — Final enable check

    If ANY check fails → return 0 → skip A/F learning update
    Result stored at FFFF78A0+3 (byte flag: CL learning active)

  0x344BA: A/F Learning loop (4 iterations)
    Iterates over 4 airflow ranges (r12 = 4, counter in r14)
    For each range:
      - Loads float from FFFF316C + offset (learning table base in RAM)
      - Calls 0xBDBCC (likely a table interpolation/update function)
      - Passes current learning value as fr14 parameter
      - Updates at 8-byte intervals (float value + metadata per range)

  0x344EE: Range validity check (4 iterations)
    Same loop structure, calls 0xBDCB6 for each range
    If ANY range returns non-zero → sets flag (r12 = 1)
    This checks if learning values are within valid limits

SECTION 5: AF 3 CORRECTION — REAR O2 TRIM (DISABLED)
================================================================================

  AF 3 Correction Limits @ 0x35FFC:
    Minimum: 0.000000 → 0.00%
    Maximum: 0.000000 → 0.00%

  BOTH LIMITS ARE ZERO. The rear O2 sensor has NO influence on fuel trim
  in this ROM. Per the XML definition:

    "CHANGING THIS VALUE MIN-MAX TO 0 INPUT RESULTS IN THE DISABLITY
     OF THE REAR O2 SENSOR INPUT ON TARGET AFR. THIS AFR TRIM 3 CAN
     CAUSE RICHER THAT DISIRED IDLES WHEN LEFT ON. DISABLE IF
     EXPERINCING RICHER THAN IDEAL IDLES."

  This is an intentional tune choice, not a defect.


================================================================================
SECTION 6: CL FUELING TARGET — WHAT AFR DOES CL AIM FOR?
================================================================================

The CL fueling target is 14.7 AFR (stoichiometric) adjusted by compensation
tables based on engine load and coolant temperature.

─── Compensation Tables ───────────────────────────────────────────────────

  CL Fueling Target Comp A (Load) @ 0xD14D0:
    3D table (RPM × Load → AFR additive adjustment)
    Typical values: -0.01 to -0.61 AFR
    At high load: up to -0.61 AFR (target = 14.09)
    At low load:  -0.01 AFR (target ≈ 14.69)

  CL Fueling Target Comp B (Load) @ 0xD1740:
    Same format, alternate calibration

  CL Fueling Target Comp Imm. Cruise (ECT) @ 0xD13B0:
    Coolant temperature compensation during cruise

  CL Fueling Target Comp Imm. Non-Cruise (ECT) @ 0xD141C:
    Coolant temperature compensation during non-cruise

  CL Fueling Target Comp (ECT) Disable @ 0xCBF9C:
    Threshold: 119.0 °F
    Above this coolant temp, ECT compensation is no longer applied.

  NOTE from XML: "Lean compensation in this table will potentially force
  open loop during normally closed loop fueling conditions."


================================================================================
SECTION 7: OL FUEL MAP SELECTION — THE FAILSAFE PATH
================================================================================

When the ECU is in Open Loop, fuel is determined by lookup tables rather
than O2 sensor feedback. Map selection depends on IAM (Ignition Advance
Multiplier):

  Primary OL Fuel Map Switch (IAM) @ 0xCC16C: threshold = 0.500

  IAM >= 0.500:  Normal maps (KCA Low @ D0244, KCA High @ D0404,
                              KCA Alternate @ CFD30)
                 Max raw = 0x2B (43) → minimum AFR = 11.00

  IAM < 0.500:   Begin failsafe transition
  IAM < 0.050:   Full failsafe

  Failsafe Map @ 0xD05C4:
    Max raw = 0x46 (70) → minimum AFR = 9.50
    At 2000 RPM / 2.60 g/rev: raw=0x3C (60)
      AFR = 14.7 / (1 + 60 × 0.0078125) = 14.7 / 1.46875 = 10.02
    THIS IS THE ROOT CAUSE OF THE OBSERVED "10.02 AFR Final Fueling Base"

  Minimum Active Primary OL Enrichment @ 0xCC170: 0.050 (5%)
    Ensures at least 5% enrichment when in OL mode.

  Code at 0x3605E (dispatched from 0x4A0F4):
    Reads IAM from RAM[FFFF3234]
    Compares against threshold at CC16C
    Selects appropriate fuel map set
    Multiple descriptor pointers at AD674, AD690, AD6AC, AD6C8, AD6E4, AD700
    point to the various OL fuel map descriptors


================================================================================
SECTION 8: THE SHORT-TERM FUEL TRIM / AFC MECHANISM — FULLY TRACED
================================================================================

The AFC (short-term fuel correction) has been traced at the instruction level.
It is called from the CL fueling dispatcher at 0x33304, which sequences
through a chain of BSR calls to process the complete CL correction pipeline.

─── AFC Dispatcher: Function 0x33304 ───────────────────────────────────────

  This function is the CL fueling master controller. After checking
  CL enable conditions (RAM 0xFFFF65F0, 0xCBBD8-0xCBBDB), it calls:

  1. BSR 0x33D1C  — cl_fuel_target_B    (CL Fueling Target Comp B/Load)
  2. BSR 0x33CC0  — cl_fuel_target_A    (CL Fueling Target Comp A/Load)
  3. BSR 0x33658  — afc_sensor_prep     (sensor state & conditioning)
  4. BSR 0x33FCE  — afc_target_calc     (target computation w/ compensation)
  5. BSR 0x340A0  — afc_correction_acc  (correction factor accumulator)
  6. BSR 0x342A8  — afc_pi_controller   (THE AFC PI LOOP)
  7. JSR 0x3439E  — afc_enable_gate     (AFC enable/disable check)
  8. JSR 0x343CE  — afc_output_clamp    (AFC output limiter)

  Arguments are passed via R4/R5 pointers into a RAM variable address
  table at 0x63A2C-0x63A70 which maps correction slots:
    0x63A2C → 0xFFFF77C8  (CL correction state — widely consumed)
    0x63A3C → 0xFFFF782A  (AFC intermediate value)
    0x63A40 → 0xFFFF7864  (AFC correction output)
    0x63A44 → 0xFFFF77DC  (CL target comp A output)
    0x63A48 → 0xFFFF77E0  (CL target comp B output)

─── AFC PI Controller: Function 0x342A8 ────────────────────────────────────

  This is the core short-term correction loop. Disassembly reveals:

  RAM variables read:
    0xFFFF77C8  — Current CL correction state (accumulated)
    0xFFFF6540  — Sensor state variable
    0xFFFF7448  — CL/OL mode flag (shared with learning + scheduler)

  Operations (from disassembly):
    1. Load current correction state from 0xFFFF77C8
    2. Branch based on CL conditions (bf/s at 0x342C2, 0x342CA, 0x342D4)
    3. Compute error: fsub FR8,FR4 (actual - target) at 0x342DA
    4. Look up P gain: jsr 0xBEAB0 (table lookup) at 0x342E0
    5. Compare vs limit: fcmp/gt FR0,FR8 from ROM 0xCC000 (=2.0)
    6. Conditional clamp at ROM 0xCC004 (=20.0 max %)
    7. Store intermediate to RAM 0xFFFF7864 at multiple branch paths
    8. Second error path: fadd FR8,FR4 at 0x34374 (reverse direction)
    9. Second gain lookup: jsr 0xBE970 at 0x34378
    10. Compare vs ROM 0xCC008 (=0.0) and 0xCC00C (=1.0)
    11. Final store to RAM 0xFFFF7864

  The function has 6 distinct branch paths writing to 0xFFFF7864,
  handling: lean error, rich error, limit clamping, and enable/disable.

─── AFC Calibration Constants (UNDOCUMENTED — not in XML) ──────────────────

  ROM Address   Float Value   Purpose
  ──────────────────────────────────────────────────────────────────────
  0xCC000       2.0           AFC P gain / step size
  0xCC004       20.0          AFC max correction limit (%)
  0xCC008       0.0           AFC min correction or dead band
  0xCC00C       1.0           AFC I gain / unity scale factor
  0xCC010       0.0           AFC enable threshold (gate, in 0x3439E)
  0xCC014       200.0         AFC output upper clamp (in 0x343CE)
  0xCC018       190.0         AFC output lower clamp
  0xCC01C       0.0           Additional limit / reserved

─── AFC Enable Gate: Function 0x3439E ──────────────────────────────────────

  Short function. Calls 0x2BE2C and 0x2BE38 (utility checks).
  Reads ROM 0xCC010 (=0.0). This gates whether the AFC correction
  is applied. When conditions aren't met, AFC output is zeroed.

─── AFC Output Clamp: Function 0x343CE ─────────────────────────────────────

  Reads output from RAM 0xFFFF7820, applies:
    - Upper clamp from ROM 0xCC014 (=200.0%)
    - Lower clamp from ROM 0xCC018 (=190.0%)
    - Additional logic from ROM 0xCC01C (=0.0)
  Stores final clamped AFC back to RAM 0xFFFF7820.

  Also reads 0xFFFF64D8, 0xFFFF79C2, 0xFFFF79C4.

─── CL/OL AFC Decision & Hysteresis: Function 0x033DBE ─────────────────────

  This function decides whether AFC runs. Sets GBR = 0xFFFF77F4.
  Calls 0x22F92 for CL enable check. When CL is active, it looks up
  a correction from table descriptor 0xACE8C and stores to 0xFFFF77F4.
  When CL is inactive, it writes 0.0 (AFC disabled).

  Hysteresis thresholds at ROM 0xCBFD0-0xCBFE4 control the switching
  between CL active/inactive states to prevent oscillation.

─── AFC PI Output Stage: Function 0x340A0 ──────────────────────────────────

  This is the PI controller output stage with full P and I gain
  table lookups. It computes the blended PI correction:

  PROPORTIONAL path:
    P_gain_A = table lookup from 0xACEA0 (indexed by load)
    P_gain_B = table lookup from 0xACEB4 (indexed by RPM)
    P_norm   = table lookup from 0xAC4FC (indexed by airflow)
    proportional = (P_gain_A + P_gain_B) * P_norm

  INTEGRAL path:
    I_comp   = 2D table from 0xAD928 (indexed by coolant × AFR)
    I_gain_A = table lookup from 0xACEC8 (indexed by load)
    I_gain_B = table lookup from 0xACEDC (indexed by RPM)
    I_norm   = table lookup from 0xAC510 (indexed by airflow)
    integral = (I_comp + I_gain_A + I_gain_B) * I_norm

  BLEND (classic PI):
    output = proportional * alpha + integral * (1 - alpha)

  Reads flags at: 0xFFFF8E7E, 0xFFFF85D7
  Outputs to: 0xFFFF7870, and via subroutine bsr 0x33A0E

─── Where AFC Feeds Into IPW ───────────────────────────────────────────────

  The AFC correction value in RAM 0xFFFF77C8 is consumed by the
  fuel correction accumulator function at 0x320AE. This function:

  Reads:
    0xFFFF77C8  — AFC/CL correction (from the PI controller)
    0xFFFF7820  — AFC output (clamped value)
    0xFFFF77D8  — Additional correction factor
    0xFFFF781C  — Additional correction factor
    0xFFFF77E4  — Additional correction factor
    0xFFFF77DC  — CL target comp A output
    0xFFFF7448  — Mode flag
    0xFFFF7714  — IPW-related variable
    0xFFFF7710  — IPW-related variable

  Performs table lookups via 0xBE608, 0xBE628, 0xBE970, 0xBE5A8
  to combine all correction factors (AFC, LTFT, enrichments, etc.)
  into the final injector pulse width multiplier.

  This is the final stage where:
    Final IPW = Base IPW × (1 + AFC) × (1 + LTFT) × other corrections

─── Why AFC and AFL are zero during OL→CL transitions ────────────────────

  Now confirmed by code trace:

  a) CL→OL is IMMEDIATE (delay = 0 in this ROM, see 0xCBC62)
  b) During OL, the AFC dispatcher (0x33304) does NOT execute the
     PI controller — the BSR chain is gated by the CL enable check
  c) AFC output (0xFFFF7864) is zeroed or frozen during OL
  d) On return to CL, the PI controller restarts from the current
     state — likely near zero if OL was long enough
  e) The front O2 sensor smoothing adds additional lag
  f) If the transition is brief (bouncing at the threshold), AFC
     never has time to build up a meaningful correction

  The A/F Learning Max Threshold (Alpha table) controls whether LTFT
  carries into OL. If set below the D range airflow, learnings are
  completely excluded from OL fueling — the OL maps run "raw."
