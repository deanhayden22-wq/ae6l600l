7. REGION CHARACTERIZATION
==============================================================================

Primary subsystem: OBD-II DIAGNOSTIC MONITORS

This region is almost entirely dedicated to on-board diagnostic (OBD-II)
monitor functions. The evidence is overwhelming:

  a) Three named diagnostic task entry points (task53, task56, task58)
  b) 35 calls to dtc_set_pending, 31 to dtc_clear
  c) 21 calls to check_diag_state, 20 to diag_precondition_check
  d) 36 calls to check_engine_running (precondition for monitors)
  e) Massive use of descriptor-based calibration reads (114x desc_read_float)
  f) Heavy uint8_pack/uint16_pack usage for diagnostic result output
  g) RAM workspace concentration in FFFF90xx-FFFF96xx (diagnostic state)
  h) RAM output concentration in FFFF24xx (diagnostic output buffer)

The region implements the following diagnostic monitor types:
  - Diagnostic readiness computation (task53, 0x0602DC)
  - EVAP purge system diagnostic (task56, 0x066580)
  - MAF sensor diagnostic (task58, 0x06F0B8)
  - VVT error feedback monitoring (0x0627D8)
  - Multiple unnamed sensor/actuator diagnostic monitors
  - DTC management helpers (set/clear/check patterns)
  - Batch workspace serializers (pack diagnostic results)

Function templates observed:
  1. MONITOR ENTRY: check_diag_state -> check_engine_running ->
     diag_precondition_check -> [monitor logic] -> dtc_set_pending/dtc_clear
  2. CAL LOADER: rtos_workspace_init -> N x desc_read_float -> RTS
  3. RESULT PACKER: load uint16_pack/uint8_pack ptr -> N x JSR with
     sequential descriptor indices -> RTS
  4. STATUS STUB: load RAM addr -> read/write -> RTS (2-4 instructions)

9. KEY FINDINGS
==============================================================================

a) The 0x064100-0x065D00 "float_data" block is MISCLASSIFIED in
   rom_region_map.txt. It is a structured descriptor dispatch table with
   12-byte entries. Each entry contains a function pointer (0x0640F4) and
   two sequential calibration addresses in the 0x0D92xx range. This table
   likely drives a parametric diagnostic monitor framework.

b) The 10 functions called from the 0x048234 dispatch chain are NOT
   related to fuel pump control despite appearing in fuel_pump_raw.txt.
   They are diagnostic monitor subroutines sharing a scheduler timeslot.

c) The DTC helper cluster at 0x06AA00-0x06BB00 contains the bulk of
   DTC state management logic. Many functions are tiny accessor stubs
   (4-8 instructions) wrapping uint16_pack/unpack calls.

d) RAM workspace FFFF24xx (183 references) is the primary diagnostic
   output buffer. FFFF90xx-FFFF96xx (534 refs combined) hold diagnostic
   monitor state and intermediate results.

e) The function at 0x0640F4 (inside the "data" block) may be a generic
   parametric monitor handler invoked via the dispatch table. It is not
   listed as a Ghidra function, suggesting Ghidra missed it.
